# gitos-lenses — quant-discipline lens pack

A **frozen, portable** set of gitos [lenses](https://github.com/wizzo-gmb/gitos) — operator-imported
*approach* context the four gitos roles fold into their work on demand, within their boundary (engine
v11 / WO-023). This pack distills the transferable **quantitative-research discipline** from private
research repositories into six lenses. It is a one-time distillation: **no sync, no versioning back to
source.**

## The six lenses

See `index.md` for the registry. In short:

1. **quant-validation-discipline** — don't trust a signal until it survives stationarity + purged/embargoed
   OOF + exposure-preserving controls + permutation significance, with parameters locked before new data
   and one null per layer.
2. **causal-deployment-ordering** — environment selection → structural diagnostics → causal (entry-frozen,
   lagged) → costed deployment (gross → base-cost net → stress-cost net); only what survives the trading
   constraints is "operational".
3. **regime-aware-modeling** — multi-state (not binary) regimes; hedge = measured variance compression;
   switch on confirmed shifts with an explicit lag + hysteresis band.
4. **fragility-variance-response** — `R_excess = α − β·σ²` fragility; coupled-pair signed densities with
   asymmetric entry/exit; put-call parity as an independent gate.
5. **evidence-reproducibility-reporting** — Evidence Map with reconstructable claim chains,
   headline-numbers-as-descriptors (full gross → base-cost net → stress-cost net ladder), reproducibility
   boundary, interpretation guardrails.
6. **quant-foundations** (light) — a method-awareness map from pricing through vol surfaces, GARCH, and
   entropy/readability to allocation: name the assumption each method carries and check it holds.

## How to use them

In any gitos repo running engine **v11+**:

1. Copy the lens file(s) you want into that repo's `.gitos/agents/`.
2. Append their rows to that repo's `.gitos/agents/index.md`.
3. That's it — the four roles read a matching lens when their work matches its `domain` / `when-to-apply`,
   and apply it **within their role's boundary** (a read-only role stays read-only).

The pack is copy-from-here; nothing syncs back. Edit a lens freely in the repo you drop it into.

## What these are (and are not)

- **Approach, not facts.** A lens is *how to work a domain*. Specific numbers, tickers, and results belong
  to a project's brain, not here.
- **Approach, not code.** A lens may describe a method; it never embeds a strategy's code.
- **Distilled + normalized** into the gitos idiom. Each lens's `source` field records the *kind* of
  discipline it came from — not the private repositories, files, or results behind it.
